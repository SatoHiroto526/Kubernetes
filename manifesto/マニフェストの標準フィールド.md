# マニフェストの標準フィールド
## 概要
- APIバージョン（apiVersion）
```
オブジェクト操作の際に使うKubernetes APIのバージョン。
```

- オブジェクト種別（kind）
```
DeploymentやServiceなど
```

- metadata
```
オブジェクトのメタデータを記載。オブジェクトを識別するためのnameやUIDなど。
```

- spec
```
オブジェクトのあるべき姿を記載する。オブジェクトによってはspecを記載しないものもある。
```

## APIバージョン記載方法例
- appsグループ（DeploymentやReplicaSetなど）
```
apiVersion: aps/v1
```

- networking.k8s.ioグループ（Ingress）
```
apiVersion: networking.k8s.io/v1
```

・coreグループ（PodやServiceなど）
```
apiVersion: v1
```

※apiVersionとkindの組み合わせでオブジェクト種別とバージョンが一意に定まる。  
※apiVerisonとkindのセットはKubernetesオブジェクトにとって特別な意味を持つことから「GVK（Group、Version、Kind）」と呼ばれる。  
※GVKにmetadataが加わると、クラスタ内のKubernetesオブジェクトを一意に特定することができる。  
※specにはKubernnetesオブジェクトのあるべき姿を記載し、フォーマットはGVKごとに異なる。

## クラスタがサポートするAPIバージョン確認方法
### 各オブジェクトのAPIバージョン確認
```
kubectl explain <オブジェクト名>
```
#### 例）PodのAPIバージョンを確認
```
kubectl explain pod
```

### KubernetesクラスタがサポートするAPIバージョン確認
```
kubectl api-resources
```

## 標準フィールド記載例
### Podの場合
```
apiVersion: v1
kind: Pod
metadata:
    # nameはイメージ識別子
    # コンテナでいうところのlocalhost/api:v1みたいなもの
    name: api
    # 任意に設定できる Key-Value ペア。以下用途で使用。
    # Pod のグルーピング：同じラベルを持つ Pod をまとめて管理。
    # Serviceとの紐づけ：erviceのselectorapp: api-server を指定すると、そのラベルを持つPodにトラフィックを送れる。
    # 監視・ログ管理：PrometheusやFluentdで、ラベルに基づいてメトリクスやログを集約。
    labels: 
        # 以下の場合、"app"がキー、"web-api"がバリュー
        app: web-api
spec:
    containers:
    # あるべき姿を記載
```
